## 1. What we need

Mahima's Product Performance Report (live at `digital-marketing-member-pages.vercel.app/pages/mahima.html`) requires two columns per product, per Google Ads campaign:

| Column | Meaning (as shown in the Google Ads UI, Products tab) | Example values |
|---|---|---|
| **Status** (feed/eligibility status) | Whether the product is eligible to serve in ads | `Eligible`, `Not eligible in any campaigns`, `Product paused`, `Disapproved` |
| **Missing Attribute** | Which required/recommended feed attributes the product is missing | e.g. `gtin`, `color`, `size`, or empty if none |

Both are per-product diagnostics generated by **Google Merchant Center** (the same data visible in the Google Ads UI under Products → Status / Issues columns, and in Merchant Center under Products → Diagnostics).

Scope: all products in the ledsone.de Merchant Center account (4 merchant IDs: **274357352, 5351990695, 5349761386, 5348836557**). Mahima's report alone covers 2,370 distinct product IDs across 5 campaigns.

## 2. Verification performed — this data does NOT currently exist in PostgreSQL

Re-verified on **2026-07-09** (second full pass; first pass 2026-07-08). Method: searched `information_schema.columns` across **all ~40 user schemas** for every column matching `feed_status`, `feed_health`, `eligib%`, `missing_attribute%`, `merchant_center%`, `disapprov%`, `product_status`, `item_status`, `approval%`, `gmc%`, `diagnostic%`, plus a second pass on generic `status`/`issue`/`flag`/`notes` columns cross-referenced against tables that also carry a product/item ID column. Every candidate table was then opened and row-counted.

### 2a. The intended destination table exists but is EMPTY

**`raw_data.gmc_product_diagnostics_daily` — 0 rows.**

The schema is already correctly designed for exactly this data:

```
id, merchant_id, date, sku, item_id, product_title,
destination_status, approval_status, issue_severity, issue_code,
issue_description, affected_country, source_file, row_count,
payload_hash, import_timestamp, audit_run_id, validation_status, created_at
```

The table was clearly created for a GMC diagnostics import pipeline that **was never run (or never built)**. Nothing has ever been loaded into it.

### 2b. Corroborating evidence the pipeline never ran

`staging_ai.pmax_18m_postmortem_fact_v1` (238,576 rows — a real, populated fact table) carries a `gmc_diagnostics_rows` join-count column. It is **0 for every one of the 238,576 rows**, confirming that when that table was built, its join against the diagnostics source found nothing — consistent with the source being empty, not with a join bug.

### 2c. Every other candidate checked — all are tiny audit/pilot samples or unrelated systems

| Table | Rows | Verdict |
|---|---|---|
| `staging_ai.cppc_pmax_campaign_digital_twin_rows_v1` (has `feed_status`, `merchant_center_issue`) | 193 (only 4 distinct products) | Sample/prototype data, not a catalog |
| `staging_ai.cppc_google_pilot_input_output_audit_v1` (has `feed_status`, `eligibility_*`) | 20 | Pilot audit sample |
| `staging_ai.cppc_google_pmax_readiness_guardrail_v1` (`feed_status`) | 20 | Audit sample |
| `staging_ai.pmax_baseline_kpi_audit` (`feed_status`) | 11 | Audit sample |
| `staging_ai.cppc_pmax_campaign_passport_v1` (`merchant_product_status`) | 47 | Campaign-level, not per-product |
| `staging_ai.cppc_pmax_campaign_digital_twin_health_v1` (`feed_health_score`) | 3 | Campaign-level aggregate |
| `staging_ai.cppc_pmax_domain_enrichment_opportunity_v1` / `..._enrichment_gap_v1` / `v_pmax_campaign_domain_signals_v1` (`missing_attribute_count`, `has_disapproved`) | 95 each | Domain-level aggregates, not per-product |
| `staging_ai.customer_requirement_extraction_pilot` (`missing_attributes_json`) | 100 | Unrelated (customer-service pilot) |

`public.google_merchant_products` (≈498k rows) was also checked: it holds the feed **content** (title, price, availability, feed_label, etc.) but has **no diagnostics columns** (no status/issues/approval fields) — it cannot supply this data.

### 2d. The database's own audit view already documents this exact gap

`staging_ai.v_pmax_18m_feed_failure_v1` (238,576 rows) — an internal feed-failure analysis view — carries the literal failure reason **`GMC_DIAGNOSTICS_SOURCE_EMPTY`** on its rows, with root cause: *"Feed readiness cannot be fully proven where GMC diagnostics are empty, or required Merchant Center fields are weak/missing."* The system itself has already flagged the missing ingestion.

### 2e. Third-pass completeness checks (2026-07-09)

- All **json/jsonb columns** database-wide (~130) enumerated — none holds GMC diagnostics (they are Wayfair raw payloads, IPC configs, evidence blobs for unrelated engines, etc.)
- All **43 tables named** `%google%`/`%gmc%`/`%merchant%`/`%shopping%` enumerated; every one with real row counts opened and inspected (e.g. `cppc_google_ads_product_history_v1`, 80,850 rows = SKU/month spend history only, no status fields)

**Conclusion: the data does not exist anywhere in PostgreSQL, at any scale — verified across three passes, and independently confirmed by the database's own audit view (`GMC_DIAGNOSTICS_SOURCE_EMPTY`). This is a missing ingestion pipeline, not a missing query.**

## 3. What we're asking the developer to build

Populate **`raw_data.gmc_product_diagnostics_daily`** (schema already in place) with a daily import from the Google Merchant Center **Content API for Shopping**, `productstatuses` resource, for all 4 ledsone.de merchant IDs.

The API's `productstatuses.list` response maps almost 1:1 onto the existing table:

| API field | Table column |
|---|---|
| `productId` / offer id | `item_id` / `sku` |
| `title` | `product_title` |
| `destinationStatuses[].status` (destination `Shopping ads`) | `destination_status` / `approval_status` |
| `itemLevelIssues[].code` | `issue_code` |
| `itemLevelIssues[].description` | `issue_description` |
| `itemLevelIssues[].severity` | `issue_severity` |
| `itemLevelIssues[].applicableCountries` | `affected_country` |

Missing-attribute information arrives as item-level issues with codes like `missing_required_attribute` / attribute-specific codes — no extra endpoint needed.

**Requirements:**
1. Daily scheduled run (matching the cadence of `ppc_etl_performance_data`, which updates daily ~07:03).
2. Cover all 4 merchant IDs listed in §1.
3. One row per product × issue (or one row per product with null issue fields if fully approved) per day, so history accumulates.
4. Populate `import_timestamp`, `source_file`/`audit_run_id` for traceability, per the columns already present.

**Access needed (if not already held by the ETL system):** Google Cloud project with Content API for Shopping enabled + OAuth/service-account access granted to the 4 Merchant Center accounts. (The existing `google_merchant_products` importer already reads feed content from these accounts, so credentials likely already exist — the diagnostics endpoint just needs to be added to that same pipeline.)
