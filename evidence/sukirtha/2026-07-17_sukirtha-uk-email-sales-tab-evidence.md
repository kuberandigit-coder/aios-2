# Sukirtha — UK Email-Sales Tab — Evidence

**Date:** 2026-07-17
**Purpose:** Add a third staff tab to the shared sales dashboard (`reports/digital-marketing-member-pages/pages/sales.html`) for Sukirtha, who owns email marketing across ledsone.co.uk (UK) and ledsone.de (DE). This task built **UK only**; DE is a scaffolded placeholder for a later task.

## Business Question
"What sales came from the email channel, and how much of that traffic did Sukirtha's email marketing actually drive to conversion?"

## Critical difference from Kamsi / Dilaksi
Kamsi and Dilaksi's dashboards are **product-scoped**: only orders containing a product from their specific allocation CSV count, and only the matched line items within those orders. Sukirtha's dashboard is **store-wide**: every single order in the shop counts, regardless of which products are in it. There is **no product-allocation CSV and no line-item-matching step** for Sukirtha — every line item on every qualifying order is included in full.

## Definition (mirrors Kamsi's Fully Organic / First-Session Organic pattern, but for EMAIL)
Two groups only, shown exclusively — no other buckets:
- **Fully Email**: every touchpoint in the order's `customerJourneySummary` (first visit, last visit, and every visit in between) classifies as `EMAIL`, using the identical `classifySession()` logic already used in `api/sales-kamsi.js` (email evidence = `sourceType === 'NEWSLETTER'` OR `medium === 'email'` OR source/sourceDescription containing "email").
- **First-Session Email**: the order's FIRST visit classifies as `EMAIL`, but the journey is not fully email (at least one other touchpoint is a different channel).

Orders that qualify for neither group are excluded from Sukirtha's totals entirely — no Direct, Referral, No Journey Data, AI Tools, Organic, Paid, or Social sections on this tab (those belong to other staff/other dashboards).

## What was built
1. **`reports/digital-marketing-member-pages/api/sales-sukirtha-uk.js`** (new serverless function)
   - Reuses `classifySession()`, `classifyOrderJourney()` (adapted: `FULLY_EMAIL` / `MIXED_JOURNEY` / `NON_EMAIL` in place of the organic variants), `hasPaidEvidence()`, `SEARCH_ENGINES`, the Shopify GraphQL order-fetching logic (`fetchOrdersForMonth`), the documented `created_at` date-only-boundary padding fix, `resolveReportMonth`, `SUPPORTED_MONTHS`, the static-snapshot caching pattern, and `amt`/`ccy`/`round2` helpers — all copied faithfully from `api/sales-kamsi.js`.
   - Does **not** reuse `matchLineItemToKamsi` or any CSV loading — `buildSukirthaOrderRow()` includes every line item on every order with no product filter.
   - Response payload: `summary` (fullyEmailOrders count + stats), `firstSessionEmailSummary`, `combinedSummary` (Fully Email + First-Session Email rows summed), `allSukirthaOrders` (both buckets, each row tagged `group: 'Fully Email'` or `'First-Session Email'`), `meta`.
   - Serves static snapshots from `api/data/sukirtha-uk-sales-<month>.json` when not force-refreshing; same 55s in-memory cache pattern as Kamsi/Dilaksi.
2. **6 monthly static snapshots** generated by deploying to Vercel production and curling `/api/sales-sukirtha-uk?month=2026-0X&refresh=1` for X=1..6 (env vars are Production-scoped Sensitive vars that don't resolve locally, per the Dilaksi precedent). Saved to `api/data/sukirtha-uk-sales-2026-0[1-6].json`.
3. **Sukirtha tab added to `sales.html`**:
   - New top-level tab button "Sukirtha · Email" (`tabBtnSukirtha` / `tabSukirtha`), and `showStaffTab(n)` updated to handle `'sukirtha'` alongside `'kamsi'`/`'dilaksi'` — explicitly checked to avoid the blank-tab regression noted in the Dilaksi evidence file.
   - UK / DE sub-tab row (`reqtabs`/`reqtab` style) via `showSukirthaRegion(r)`. Only UK is wired to real data; DE shows a static "Coming soon" placeholder — no fetch, no fabricated data.
   - UK sub-tab: month tabs Jan–Jun 2026, top-right styled Export CSV button (group-summary export, `class="primary"` inside `.exportbar`, no line-item-table toolbar CSV button — matches current final Kamsi/Dilaksi pattern), KPI cards (Total Sukirtha UK Orders, Units Sold, Gross Sales, Refunds, Net Sales, Unique Products, Data Retrieved At), explanatory sub-text stating the two-group definition and the store-wide (not product-scoped) nature, group-breakdown table (2 rows: Fully Email, First-Session Email — clickable to filter), line-item order table (search by order name / product ID, Group filter dropdown, session-detail expandable rows), footnotes section.
   - `S`-prefixed globals/functions throughout (`S_DATA`, `S_CURRENT_MONTH`, `sSelectMonth`, `sLoad`, `sRenderAll`, `sGroupOf`, `scFlatRows`, `scFilteredRows`, `scGroupBreakdown`, `scRenderAll`, `scRenderTable`, `scExportCsv`, `sGroupExportCsv`) — grepped the whole file for `function s[A-Za-z]*` / `function d[A-Za-z]*` / `function k[A-Za-z]*` and confirmed zero name collisions (the only duplicate function name in the file, `downloadCsv`, is a pre-existing duplicate shared by the Kamsi and Dilaksi blocks already, not something introduced here).
